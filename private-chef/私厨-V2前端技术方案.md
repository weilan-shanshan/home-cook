# 私厨 V2 前端技术方案

> 最后更新：2026-04-15  
> 输入来源：`私厨-V2产品方案.md` + `私厨-技术方案.md` + 当前 `frontend/` 实现  
> 适用范围：`private-chef/frontend/` 的 V2 页面、路由、状态管理、交互与实现约束

---

## 一、目标

V2 前端目标不是推翻现有 React PWA，而是在现有基础上完成一次 **“订单中心化”升级**：

1. 把“订单”提升为一级导航。
2. 把下单后的承接页从弱反馈变成明确的订单结果页/订单详情页。
3. 把菜品创建、图片上传、订单追踪、评论分享、收藏复购串成连续体验。
4. 为后续点赞、成就、分享卡提供稳定的页面与状态骨架。

---

## 二、前端总体方案

### 1. 技术栈延续

继续沿用当前技术栈：

- React 18
- Vite 5
- React Router 7
- TanStack Query 5
- shadcn/ui
- Tailwind CSS

### 2. 前端架构原则

1. **路由表达产品结构**：一级导航对应一级路由。
2. **订单详情是核心承接页**：下单成功、通知跳转、评论分享都归到订单详情。
3. **Query key 以领域划分**：recipes / orders / comments / achievements / notifications。
4. **优先服务端真实状态**：不做复杂本地业务状态机，客户端只做必要乐观更新。
5. **上传体验分离**：菜品保存成功与图片上传状态分开呈现。

## 四、路由重构方案

## 1. 当前路由现状

当前 `App.tsx` 已有：

- `/`
- `/orders`
- `/order/create`
- `/wishes`
- `/favorites`
- `/profile`
- `/recipe/*`

### 存在问题

1. 一级导航入口是 `/order/create`，强调“动作”而不是“结果”。
2. 缺少 `/orders/:id` 对应的订单详情承接页。
3. 许愿仍占用一级导航位置。

## 2. V2 路由建议

```text
/                          首页
/menu                      点菜首页
/menu/create-order         下单确认页
/orders                    订单列表页（进行中 / 已完成）
/orders/:id                订单详情页
/profile                   我的页
/favorites                 收藏页
/wishes                    许愿页（次级）
/recipe/new                创建菜品
/recipe/:id                菜品详情
/recipe/:id/edit           编辑菜品
/comments                  评论中心（P1）
/achievements              成就页（P2）
/share/order/:id           分享卡预览页（P1）
```

## 3. 兼容策略

- 保留 `/order/create` 一段时间，前端跳转到 `/menu/create-order`
- 保留 `/orders` 不变，新增 `/orders/:id`
- 旧 TabBar 的“点餐”直接切到 `/menu`

## 五、页面实现方案

## 1. 首页 HomeV2

### 页面目标

- 承接回访
- 呈现家庭动态
- 让用户快速进入点菜、订单、收藏

### 模块组成

1. 今日推荐
2. 常点菜
3. 家人最近点了什么
4. 家人最近做了什么
5. 最近评论 / 暖心互动
6. 家庭成就卡片
7. 快捷入口

### 数据来源

- `GET /api/home/summary`（V2 建议新增聚合接口）
- 或前期并行查询：recipes + orders + cookLogs + achievements

---

## 2. 点菜页 MenuPage

### 页面目标

- 替代当前“直接进入下单表单”的入口方式
- 先看菜，再下单

### 模块组成

1. 搜索框
2. 标签筛选
3. 常点菜区
4. 收藏优先区
5. 菜品列表
6. 当前待下单清单（浮层/底部抽屉）

### 关键交互

- 点菜不是立刻提交订单，而是先加入待下单清单
- 待下单清单统一进入确认页
- 菜品卡支持：查看详情 / 收藏 / 快速加单

---

## 3. 下单确认页 CreateOrderV2

### 页面目标

- 统一完成餐次、日期、备注确认
- 提交成功后进入订单详情

### 表单字段

- meal_type
- meal_date
- note
- items[]

### 提交后行为

```text
提交成功 → toast + invalidate(['orders']) + navigate(`/orders/${id}`)
```

### 与当前差异

当前 `useCreateOrder` 只做列表刷新，V2 要补：

- 进入订单详情
- 成功态文案
- “继续点菜 / 查看订单”后续动作

---

## 4. 订单列表页 OrderListV2

### 页面目标

- 让订单成为一级信息空间
- 明确“进行中 / 已完成”分区

### 页面结构

```text
订单页
├─ Tabs: 进行中 / 已完成
├─ 筛选：日期 / 餐次 / 点单人 / 做菜人（后两项可 P1）
└─ 订单卡片列表
```

### 卡片字段

- 菜品摘要
- 谁点的
- 谁做的
- 餐次
- 日期
- 当前状态
- 评论状态
- 收藏状态
- 再来一单按钮（已完成）

### 当前页面改造点

现有 `OrderList.tsx` 已有列表基础，但要改成：

- 按状态主分组，不只按日期分组
- 卡片点击进入 `/orders/:id`
- 已完成卡片增加复购、评论、分享入口

---

## 5. 订单详情页 OrderDetailV2

### 页面目标

- 承接订单全生命周期
- 作为评论、点赞、收藏、分享的操作中心

### 页面结构

1. 订单头部信息
2. 状态时间线
3. 菜品列表
4. 点单人 / 做菜人信息
5. 订单备注
6. 过程评论区
7. 完成后评价区
8. 快捷操作区
   - 点赞
   - 收藏菜单
   - 分享
   - 再来一单

### UI 状态

- loading
- not found
- unauthorized
- empty comments
- completed but not reviewed

### 关键交互

- 更新订单状态后局部刷新详情与列表
- 评论成功后局部插入评论流
- 分享操作打开分享卡预览页或弹层

---

## 6. 菜品创建 / 编辑页 RecipeFormV2

### V2 核心改造点

当前问题是“菜品创建”和“图片上传”绑定太紧。

### 新交互方案

#### 阶段 1：先创建菜品实体

- 提交基础信息
- 成功后拿到 recipeId
- 明确 toast：`菜品已创建`

#### 阶段 2：独立展示图片上传状态

- 每张图独立进度
- 支持单张失败重试
- 支持离开页面后在详情页继续补图

### 页面反馈要求

- 顶部成功条：菜品已创建
- 图片区域状态：上传中 / 成功 / 失败
- 底部行动：
  - 去详情
  - 去点菜
  - 返回列表

## 七、前端状态管理方案

## 1. Query Key 规划

```ts
['recipes', filters]
['recipe', id]
['orders', filters]
['order', id]
['order-comments', orderId]
['order-reviews', orderId]
['favorites']
['home-summary']
['achievements']
['profile-summary']
```

## 2. Mutation 规划

- `createRecipe`
- `updateRecipe`
- `uploadRecipeImage`
- `retryRecipeImageUpload`
- `createOrder`
- `updateOrderStatus`
- `createOrderComment`
- `createOrderReview`
- `toggleFavorite`
- `toggleLike`

## 3. 乐观更新策略

### 可以乐观更新

- 点赞
- 收藏
- 评论列表尾部插入

### 不建议过度乐观更新

- 订单状态时间线
- 成就统计
- 分享次数

这些优先以服务端返回为准。

## 八、上传体验技术方案

## 1. 当前问题

当前 `uploadImage()` 把压缩、presign、XHR 直传打包在一起，但错误粒度太粗。

## 2. V2 拆分

前端上传状态枚举：

```ts
type UploadStatus = 'idle' | 'compressing' | 'presigning' | 'uploading' | 'saving' | 'success' | 'error'
```

### 每张图记录

- localId
- file
- previewUrl
- progress
- status
- errorMessage
- remoteUrl

## 3. UX 要求

- 失败可重试单张
- 失败不影响 recipe 本体
- 离开创建页时保留已完成结果

## 九、通知与前端回流

V2 前端不只消费 toast，还要消费“订单回流”。

## 1. 通知触达后的页面行为

- 若未来接入企微 deeplink，前端需支持：

```text
/orders/:id?from=wecom&event=confirmed
```

### 行为要求

- 打开后直达订单详情
- 可根据 query 参数滚动到对应模块（如评论区、状态区）

## 2. 站内通知准备

即使 P0 不做完整站内通知中心，前端也应预留：

- Tab 角标能力
- 顶部轻提示条
- 订单状态变化后局部提示

---

## 十、前后端契约要求

## 1. 订单详情返回结构必须扩展

V2 的 `GET /api/orders/:id` 至少需要补齐：

- order creator info
- assignee / cook info
- status timeline
- comments summary / comments list
- review summary
- like/favorite/share state

## 2. 首页聚合接口

建议新增：

- `GET /api/home/summary`

减少首页多请求拼装成本。

## 3. 分享卡接口

建议新增：

- `GET /api/orders/:id/share-card`

返回分享卡渲染数据，而不是前端自己拼所有字段。

## 十二、约束与禁止项

1. 不引入重型全局状态库。
2. 不把业务状态堆进 localStorage。
3. 不把图片上传失败等同于菜品保存失败。
4. 不在订单主链路里保留找不到结果页的跳转方式。
5. 不为了赶进度跳过 `/orders/:id` 详情承接页。

---

## 十三、完成标准

当前端 V2 核心落地后，应满足：

1. 用户能通过一级导航直接进入订单页。
2. 下单成功后必然进入订单详情或等效成功页。
3. 订单详情能承接状态、评论、分享、复购。
4. 创建菜品时，图片失败不会让用户误判“整道菜没创建成功”。
5. “我的”页能作为个人资产聚合页，而不只是家庭成员页。
