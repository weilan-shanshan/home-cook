## 一、文档定位

本文档是私厨 V2 的 **技术总方案**，用于统一前后端、数据建模、接口设计、通知机制和分期实施策略。

它解决的问题是：

- 把产品方案转译为可执行的技术架构
- 把前端方案与后端方案收敛成一份统一蓝图
- 为后续开发、任务拆分、数据库迁移和联调提供统一依据

### 与其他文档的关系

- `私厨-V2产品方案.md`：回答“做什么、为什么做”
- `私厨-V2技术总方案.md`：回答“怎么做、先做什么、前后端如何协同做”
- `私厨-技术方案.md`：保留现有基础架构、部署与技术选型基线

## 二、V2 核心目标

V2 的本质不是简单新增几个页面，而是完成一次产品主链路升级：

> **从“家庭菜谱工具”升级为“以订单为核心的家庭私厨协作产品”**

### 目标拆解

1. **订单中心化**：订单成为一级信息空间与领域核心对象。
2. **结果承接明确**：下单后必须进入订单详情或等效成功页。
3. **上传体验拆分**：菜品创建成功与图片上传状态分开表达。
4. **互动能力闭环**：订单承接评论、评价、点赞、收藏、分享。
5. **通知可靠化**：从群广播升级为订单生命周期事件触达。
6. **家庭活跃机制**：为成就、榜单、家庭动态提供数据底座。

## 2. 当前关键问题

### 产品层问题

1. 上传图片失败时，用户会误判整个菜品创建失败。
2. 创建菜品后缺少强结果承接，列表刷新感弱。
3. 下单后“我的订单”难找。
4. 通知不可靠，无法围绕订单生命周期回流。
5. 点菜 → 订单 → 评论 → 分享 → 成就没有被设计成一条连续体验链。

### 技术层问题

1. 路由层仍偏“动作导向”，缺少订单详情核心承接页。
2. 后端 `orders` 模型语义偏薄，难以承载 V2 互动能力。
3. 评论仍主要挂在 `cook_logs / ratings`，不适合订单互动。
4. 通知模块是内存队列 + webhook 文本广播，不可追踪、不可持久化。
5. 首页和“我的”缺少聚合接口支持。

## 四、V2 技术总体原则

1. **订单是主聚合根**。
2. **订单详情是前端核心承接页**。
3. **聚合优先**：后端尽量聚合输出首页与订单详情。
4. **渐进演进**：不一次性推翻现有表结构和接口。
5. **familyId 是强边界**：所有业务查询必须限定在当前家庭。
6. **上传与实体解耦**：图片失败不等于业务实体失败。
7. **通知事件化**：先落库，再投递，不再直接依赖同步广播。

## 3. 核心页面方案

### HomeV2

目标：内容分发、家庭动态、回流入口。

模块：

- 今日推荐
- 常点菜
- 最近点菜动态
- 最近做菜动态
- 最近评论 / 暖心互动
- 家庭成就卡片
- 快捷入口（点菜 / 我的订单 / 收藏 / 发布菜品）

### MenuPage

目标：从“直接表单下单”改为“先看菜、再下单”。

模块：

- 搜索
- 标签筛选
- 常点菜
- 收藏优先
- 菜品列表
- 待下单清单

### CreateOrderV2

目标：统一确认餐次、日期、备注，并在提交成功后进入订单详情。

提交后行为：

```text
提交成功 → invalidate(['orders']) → navigate(`/orders/${id}`)
```

### OrderListV2

目标：订单成为一级信息空间。

核心结构：

- Tabs：进行中 / 已完成
- 订单卡片
- 后续筛选（日期 / 餐次 / 点单人 / 做菜人）

### OrderDetailV2

目标：作为订单生命周期承接页。

结构：

- 订单基础信息
- 状态时间线
- 菜品列表
- 点单人 / 做菜人信息
- 订单备注
- 过程评论区
- 完成后评价区
- 点赞 / 收藏 / 分享 / 再来一单

### RecipeFormV2

目标：分离“菜品创建成功”和“图片上传状态”。

关键设计：

1. 先创建菜品实体
2. 再单独上传图片
3. 每张图独立状态：上传中 / 成功 / 失败
4. 单张失败可重试
5. 可在详情页继续补图

### ProfileV2

目标：从“家庭页”升级为“个人资产聚合页”。

模块：

- 我的订单
- 我的收藏
- 我的评论
- 我点过的
- 我做过的
- 家庭成就
- 家庭成员
- 通知设置

## 3. 新增 / 扩展数据模型

### 3.1 扩展 `orders`

```sql
ALTER TABLE orders ADD COLUMN cook_user_id INTEGER REFERENCES users(id);
ALTER TABLE orders ADD COLUMN completed_at TEXT;
ALTER TABLE orders ADD COLUMN cancelled_at TEXT;
```

### 3.2 订单状态轨迹表

```sql
CREATE TABLE order_status_events (
  id          INTEGER PRIMARY KEY AUTOINCREMENT,
  order_id    INTEGER NOT NULL REFERENCES orders(id) ON DELETE CASCADE,
  from_status TEXT,
  to_status   TEXT NOT NULL,
  operator_id INTEGER REFERENCES users(id),
  note        TEXT,
  created_at  TEXT NOT NULL DEFAULT (datetime('now'))
);
```

### 3.3 订单评论表

```sql
CREATE TABLE order_comments (
  id          INTEGER PRIMARY KEY AUTOINCREMENT,
  order_id    INTEGER NOT NULL REFERENCES orders(id) ON DELETE CASCADE,
  user_id     INTEGER NOT NULL REFERENCES users(id),
  role_type   TEXT NOT NULL,
  content     TEXT NOT NULL,
  created_at  TEXT NOT NULL DEFAULT (datetime('now')),
  updated_at  TEXT NOT NULL DEFAULT (datetime('now'))
);
```

### 3.4 订单评价表

```sql
CREATE TABLE order_reviews (
  id             INTEGER PRIMARY KEY AUTOINCREMENT,
  order_id       INTEGER NOT NULL REFERENCES orders(id) ON DELETE CASCADE,
  user_id        INTEGER NOT NULL REFERENCES users(id),
  score          INTEGER CHECK(score >= 1 AND score <= 5),
  taste_score    INTEGER,
  portion_score  INTEGER,
  overall_note   TEXT,
  created_at     TEXT NOT NULL DEFAULT (datetime('now')),
  UNIQUE(order_id, user_id)
);
```

### 3.5 订单点赞表

```sql
CREATE TABLE order_likes (
  order_id    INTEGER NOT NULL REFERENCES orders(id) ON DELETE CASCADE,
  user_id     INTEGER NOT NULL REFERENCES users(id),
  created_at  TEXT NOT NULL DEFAULT (datetime('now')),
  PRIMARY KEY (order_id, user_id)
);
```

### 3.6 订单分享表

```sql
CREATE TABLE order_shares (
  id          INTEGER PRIMARY KEY AUTOINCREMENT,
  order_id    INTEGER NOT NULL REFERENCES orders(id) ON DELETE CASCADE,
  user_id     INTEGER NOT NULL REFERENCES users(id),
  share_type  TEXT NOT NULL,
  channel     TEXT NOT NULL,
  created_at  TEXT NOT NULL DEFAULT (datetime('now'))
);
```

## 八、图片上传链路方案

## 1. 当前问题

当前菜品创建后立即上传图片，但错误表达粒度太粗，用户容易误解为“整道菜保存失败”。

## 2. V2 原则

- recipe 创建成功即视为业务成功
- image 保存失败只影响附件状态
- 图片上传与实体创建解耦

## 3. API 策略

- `GET /api/upload/presign`
- `POST /api/recipes/:id/images`
- `DELETE /api/images/:id`

可选新增：

- `POST /api/recipes/:id/images/retry`

## 4. 前端状态建议

```ts
type UploadStatus =
  | 'idle'
  | 'compressing'
  | 'presigning'
  | 'uploading'
```

## 十、通知系统方案

## 1. 当前问题

当前通知实现为：

- 内存队列
- 文本 webhook 广播
- 无持久化
- 无投递结果
- 无 deeplink 语义

## 2. V2 目标

将通知升级为 **订单生命周期事件驱动的可追踪投递系统**。

## 3. 数据模型

### 通知事件表

```sql
CREATE TABLE notification_events (
  id            INTEGER PRIMARY KEY AUTOINCREMENT,
  family_id     INTEGER NOT NULL REFERENCES families(id),
  event_type    TEXT NOT NULL,
  entity_type   TEXT NOT NULL,
  entity_id     INTEGER NOT NULL,
  payload       TEXT NOT NULL,
  status        TEXT NOT NULL DEFAULT 'pending',
  created_at    TEXT NOT NULL DEFAULT (datetime('now')),
  sent_at       TEXT,
  last_error    TEXT
);
```

### 通知投递表

```sql
CREATE TABLE notification_deliveries (
  id              INTEGER PRIMARY KEY AUTOINCREMENT,
  notification_id INTEGER NOT NULL REFERENCES notification_events(id) ON DELETE CASCADE,
  target_user_id  INTEGER REFERENCES users(id),
  channel         TEXT NOT NULL,
  status          TEXT NOT NULL DEFAULT 'pending',
  attempt_count   INTEGER NOT NULL DEFAULT 0,
  last_error      TEXT,
  created_at      TEXT NOT NULL DEFAULT (datetime('now')),
  updated_at      TEXT NOT NULL DEFAULT (datetime('now'))
);
```

## 4. 触发时机

- order_created
- order_confirmed
- order_preparing
- order_completed
- order_comment_created
- order_review_created（P1）

## 5. 实施策略

P0 可以先继续发群 webhook，但必须改成：

1. 先落库事件
2. 再异步投递
3. 记录失败原因
4. payload 中保留 `orderId`
5. 为未来 deeplink 预留字段

## 十四、约束与禁止项

1. 不推翻现有技术栈。
2. 不引入重型全局状态库替代 Query。
3. 不把图片上传失败等同于菜品保存失败。
4. 不继续把通知可靠性建立在纯内存队列上。
5. 不把订单评论强行塞回 `cook_logs / ratings` 语义里。
6. 不跳过 `/orders/:id` 详情承接页。
7. 不破坏 familyId 访问边界。

---

## 十五、完成标准

V2 技术主链路落地后，应满足：

1. 用户能从一级导航直接进入订单页。
2. 下单成功后必然进入订单详情或等效成功页。
3. 订单详情能承接状态、评论、分享、点赞、复购。
4. 菜品创建成功与图片上传失败能被明确区分。
5. 后端能够返回完整订单详情聚合结构。
6. 订单状态变更会留下时间线记录。
7. 通知具备最小可追踪能力。
8. 首页、我的页、成就页具备后续聚合接口基础。
