# 私厨 V2 产品方案

> 最后更新：2026-04-15  
> 适用范围：`private-chef/` 当前项目的产品升级方案  
> 说明：本文档聚焦 **产品定位、信息架构、核心流程、能力设计、分期路线图**，不替代现有《私厨-技术方案.md》的技术部署内容。

---

## 一、背景与目标

当前版本已经具备家庭菜谱、点餐、许愿、收藏、烹饪日志、基础通知等能力，但实际使用中暴露出几个关键问题：

1. **上传图片失败时体验不清晰**：用户难以判断是图片上传失败，还是整道菜创建失败。
2. **创建菜品后反馈与结果脱节**：提示成功，但列表刷新感弱，用户对“菜已经创建好了没有”缺乏确定感。
3. **点菜后我的订单不好找**：当前导航把“点餐”作为一级入口，但没有把“订单”作为一级结果入口。
4. **点完菜后企微没有可靠通知**：现有更偏群机器人广播，不是面向订单生命周期的可靠通知体系。
5. **核心高频行为没有形成闭环**：点菜 → 看订单 → 做菜/收菜 → 评论 → 分享 → 点赞/收藏 → 成就，没有被设计成连续体验。

### V2 核心目标

将产品从“家庭菜谱管理工具”升级为：

> **以订单为核心的家庭私厨协作产品**

也就是说：

- **菜谱** 是供给库
- **点菜** 是入口动作
- **订单** 是主链路中心
- **评论 / 分享 / 点赞 / 收藏 / 成就** 是活跃与复购机制

---

## 二、V2 产品定位

### 1. 用户角色

产品仍然面向同一个家庭内部（5-10人）使用，但明确区分两类高频角色：

- **点菜的人**：浏览菜、下单、追踪订单、收菜后评价、分享
- **做菜的人**：管理菜品、接单、更新状态、完成后回应评论、积累口碑和成就

### 2. 产品定位升级

V1 更像：

- 家庭菜谱 + 点餐工具

V2 更像：

- 家庭内部的“私厨协作 + 订单履约 + 轻社交”产品

### 3. V2 设计原则

1. **订单优先**：用户下单后的去向必须明确。
2. **状态透明**：订单流转必须清晰可见。
3. **轻社交**：互动要轻，不做复杂社区。
4. **低门槛分享**：分享卡片优于复杂社交关系链。
5. **家庭感**：成就体系强调温度和陪伴，不做外部竞争。

---

## 三、V2 核心问题复盘

### 问题 1：上传图片失败

#### 现状

- 菜品创建与图片上传是串行发生的，但对用户是一次“保存菜谱”的整体感知。
- 一旦图片上传失败，用户很容易误解为菜品也没创建成功。

#### V2 目标

- 把“菜品创建成功”和“图片上传状态”分开反馈。
- 支持失败后重试，不让用户重复填整张表单。

#### V2 产品要求

- 提交后先明确告诉用户：**菜品已创建**。
- 图片区域展示：上传中 / 上传成功 / 上传失败。
- 上传失败时允许单张重试。
- 菜品详情页可继续补图，而不是必须在创建页一次完成。

---

### 问题 2：创建菜品后提示成功但没刷新

#### 现状

- 技术上已做 query invalidation，但用户体感上“创建结果没有立刻回到可见列表”。

#### V2 目标

- 强化“成功落点”和“结果可见性”。

#### V2 产品要求

- 创建成功后优先进入 **菜品详情页**。
- 提供明确按钮：
  - 返回菜品列表
  - 去点菜
  - 继续补充图片
- 返回列表时，新增菜品在列表中应具备明显识别（置顶、标记“刚创建”或自动定位）。

---

### 问题 3：点菜后我的订单不好找

#### 现状

- 当前底部导航有“点餐”，没有“订单”。
- 用户完成下单动作后，缺少稳定的结果入口。

#### V2 目标

- 将“订单”提升为一级导航。

#### V2 产品要求

- 底部导航从 V1 的“首页 / 点餐 / 许愿 / 我的”升级为：

```text
首页 / 点菜 / 订单 / 我的
```

- “我的”页内继续保留“我的订单”快捷入口，形成双入口保障。

---

### 问题 4：点完菜后企微没有通知

#### 现状

- 当前通知更接近“群机器人文本广播”。
- 没有围绕订单生命周期做可靠通知和回流。

#### V2 目标

- 通知从“有事提醒一下”升级为“订单生命周期触达”。

#### V2 产品要求

- 订单创建、确认、制作中、已完成，都应有明确通知策略。
- 通知内容要能回到 **具体订单详情**。
- 支持下单人和做菜人分别收到与自己相关的通知。

---

## 四、V2 核心能力模型

V2 聚焦六类高频能力：

1. **点菜**
2. **订单**
3. **评论**
4. **分享**
5. **点赞 / 收藏**
6. **家庭成就**

这六类能力不是并列堆功能，而是围绕一个核心闭环组织：

```text
点菜 → 订单 → 履约 → 评论 → 分享 → 点赞/收藏 → 成就 → 再次点菜
```

---

## 五、V2 信息架构

## 1. 底部导航

```text
首页 | 点菜 | 订单 | 我的
```

### 导航调整说明

- **首页**：内容分发与回流
- **点菜**：发起下单动作
- **订单**：查看结果与推进履约
- **我的**：沉淀个人资产与家庭关系

> “许愿”从底部一级导航降级为首页或点菜页内入口，不再占用一级 tab。

---

## 2. 首页结构

```text
首页
├─ 今日推荐
├─ 常点菜
├─ 家人最近点了什么
├─ 家人最近做了什么
├─ 最近评论 / 暖心互动
├─ 家庭成就卡片
└─ 快捷入口
   ├─ 去点菜
   ├─ 我的订单
   ├─ 收藏
   └─ 发布菜品
```

### 首页目标

- 帮用户快速回到高频动作
- 建立家庭内部的“动态感”
- 让订单、互动、成就自然回流

---

## 3. 点菜页结构

```text
点菜
├─ 搜索
├─ 标签 / 分类
├─ 常点菜
├─ 收藏优先
├─ 菜品列表
│  └─ 菜品详情
│     ├─ 图片
│     ├─ 简介
│     ├─ 标签
│     ├─ 评论摘要
│     ├─ 点赞 / 收藏
│     └─ 加入订单
└─ 下单确认页
   ├─ 餐次
   ├─ 日期
   ├─ 备注
   └─ 提交订单
```

### 点菜页目标

- 降低点菜决策成本
- 强化“常点 / 收藏 / 再来一单”能力
- 让点菜动作尽可能短链路完成

---

## 4. 订单页结构（V2 核心）

```text
订单
├─ 进行中
│  └─ 订单卡片
│     ├─ 菜品信息
│     ├─ 谁点的
│     ├─ 谁做
│     ├─ 当前状态
│     └─ 进入订单详情
└─ 已完成
   └─ 订单卡片
      ├─ 菜品信息
      ├─ 评论状态
      ├─ 分享状态
      ├─ 收藏状态
      └─ 再来一单
```

### 订单页设计原则

- **进行中**：强调推进、提醒、动作
- **已完成**：强调沉淀、评价、分享、复购

---

## 5. 订单详情页结构（V2 最关键页面）

```text
订单详情
├─ 订单状态时间线
│  ├─ 已下单
│  ├─ 已确认
│  ├─ 制作中
│  └─ 已完成
├─ 菜品信息
├─ 点单人信息
├─ 做菜人信息
├─ 订单备注
├─ 过程评论区
├─ 完成后评价区
├─ 点赞
├─ 收藏菜单
└─ 分享
```

### 为什么订单详情是核心

订单详情页是 V2 体验闭环的承接点：

- 下单成功后跳这里
- 通知消息跳这里
- 评论、点赞、分享从这里发起
- 已完成后复购也从这里触发

---

## 6. 我的页结构

```text
我的
├─ 我的订单
├─ 我的收藏
├─ 我的评论
├─ 我点过的
├─ 我做过的
├─ 家庭成就
├─ 家庭成员
└─ 通知设置
```

### 我的页目标

- 归档个人行为资产
- 补充订单入口
- 呈现个人贡献与家庭关系

## 六、V2 核心流程设计

## 流程 1：创建菜品

```text
填写菜品信息 → 提交成功 → 生成菜品详情 → 补传图片 / 去点菜 / 返回列表
```

### 设计要求

- 菜品创建成功必须先被确认。
- 图片上传失败不阻断菜品本体存在。
- 图片失败后可在详情页继续处理。

---

## 流程 2：点菜下单

```text
浏览菜品 → 选择菜品 → 填写餐次/日期/备注 → 下单成功 → 进入订单详情
```

### 设计要求

- 下单成功后不要只 toast。
- 必须进入 **订单详情** 或 **订单成功页**。
- 成功页要给用户两个下一步：
  - 查看我的订单
  - 继续点菜

---

## 流程 3：订单履约

```text
已下单 → 已确认 → 制作中 → 已完成
```

### 设计要求

- 每个状态都能被清晰看见。
- 下单人和做菜人都能理解当前进度。
- 状态变化应触发站内/企微通知。

---

## 流程 4：订单评论

V2 不再只把评论理解为“打分”，而是分成两层：

### A. 过程评论

用于订单履约过程中交流：

- 点菜的人：想吃辣一点、少油、晚点吃
- 做菜的人：已经开始做、预计还要 15 分钟

### B. 完成后评价

用于订单完成后沉淀反馈：

- 点菜的人评价味道、分量、满意度
- 做菜的人回应反馈或补充说明

### 设计原则

- 评论主对象从“单次做菜记录”逐步过渡为“订单”。
- 保留评分能力，但评分不再是唯一评论形式。

---

## 流程 5：分享

V2 的分享不依赖复杂社交关系，而是生成适合微信生态传播的卡片内容。

### 分享类型 A：点菜分享卡

适合下单后：

- 我今天点了什么
- 包含菜单、时间、点单人昵称
- 营造期待感

### 分享类型 B：完成分享卡

适合订单完成后：

- 成品图
- 菜品信息
- 评论摘要
- 点赞/收藏数据

### 设计原则

- 用户分享的是“订单成果”而不是纯菜谱数据。
- 分享应从订单详情页一键发起。

---

## 流程 6：点赞 / 收藏

### 点赞对象

- 评论
- 完成分享卡
- 菜品成果图

### 收藏对象

- 菜品
- 常点菜单
- 分享内容 / 想再次参考的成果

### 设计原则

- 点赞负责“轻互动”
- 收藏负责“复购 / 回看 / 沉淀”

---

## 流程 7：家庭成就

成就系统不是独立玩法，而是对真实家庭行为的可视化沉淀。

### 成就维度

#### 1. 点单成就

- 点单量
- 连续点单天数
- 再来一单次数

#### 2. 做菜成就

- 做菜量
- 被点次数
- 完成订单数

#### 3. 社交成就

- 收到点赞数
- 收到收藏数
- 收到暖心评论数
- 被分享次数

### 展示方式

- 家庭总成就
- 个人成就
- 本周榜单
  - 点单王
  - 掌勺王
  - 暖心评论王
- 徽章墙

### 设计原则

- 强调家庭感、鼓励感
- 避免外部排名焦虑
- 以家庭内部温暖氛围为主

## 七、V2 能力清单

## P0：必须先解决的基础问题

1. 图片上传体验优化
2. 下单后可靠通知
3. 订单一级入口
4. 下单成功后落到订单详情

### P0 目标

2. 收藏增强（菜品 / 常点菜单）
3. 分享卡
4. 已完成订单的再来一单

### P1 目标

- 让订单完成后还有互动和复购空间

---

## P2：增强家庭活跃机制

1. 点赞
2. 家庭成就
3. 周榜 / 月榜
4. 首页家庭动态流

### P2 目标

- 提升家庭内部参与感
- 形成正反馈循环

## 九、V2 与现有技术方案的关系

V2 不推翻现有技术栈和部署架构，仍建议沿用：

- React + Vite + TanStack Query
- Hono + SQLite + Drizzle
- 腾讯云 COS 图片存储
- Cloudflare Pages + Tunnel

但在产品和数据建模层面，需要从“菜谱中心”补强为“订单中心”：

1. 路由层：补齐订单入口和订单详情承接
2. 数据层：逐步引入订单评论、订单通知、订单分享的实体或关联关系
3. 交互层：强化订单成功态、状态时间线、评论分享回流

---

## 十、V2 成功标准

如果 V2 成功，用户体验上应出现以下变化：

### 用户感知层

- 下单后马上知道去哪里看结果
- 完成后能自然评价、分享、复购
- 家里人之间的互动更顺手、更有趣

### 产品行为层

- 订单页访问频率明显提升
- 已完成订单带来更多评论、分享、再来一单
- 收藏和点赞不再只是孤立功能，而是服务复购和家庭氛围

### 系统能力层

- 通知可可靠送达并带来回流
- 评论从“偶发评分”升级为“围绕订单的互动记录”
- 成就从展示数据升级为家庭参与机制

---

## 十一、结论

私厨 V2 的关键不是再加几个零散功能，而是完成一次主链路升级：

> **从“家庭菜谱工具”升级为“以订单为核心的家庭私厨协作产品”**

核心逻辑应明确为：

```text
菜谱提供选择 → 点菜形成订单 → 订单推动履约 → 履约后产生评论/分享/点赞/收藏 → 家庭成就沉淀 → 再次点菜
```

只要把这个闭环真正建立起来，产品就会从“能用”变成“愿意持续用”。
