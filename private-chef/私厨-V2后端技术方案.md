# 私厨 V2 后端技术方案

> 最后更新：2026-04-15  
> 输入来源：`私厨-V2产品方案.md` + `私厨-技术方案.md` + 当前 `backend/` 实现  
> 适用范围：`private-chef/backend/` 的 V2 数据模型、API、通知、聚合查询与演进策略

---

## 一、目标

V2 后端目标是在保持现有 Hono + SQLite + Drizzle 架构不变的前提下，把系统从“菜谱 + 基础点餐接口”升级为 **订单中心的领域模型**。

核心方向：

1. 让订单成为主聚合根。
2. 支持订单详情、订单状态时间线、订单评论、订单评价、订单分享。
3. 将通知从群广播升级为订单生命周期事件触达。
4. 为首页聚合、家庭成就、复购入口提供查询能力。

---

## 二、后端总体架构

### 1. 技术栈延续

继续沿用：

- Hono
- better-sqlite3
- Drizzle ORM
- zod
- COS 预签名上传

### 2. 架构原则

1. **订单是核心领域对象**。
2. **保持 familyId 鉴权边界**：所有查询都必须落在当前家庭内。
3. **聚合优先于多次拼接**：订单详情、首页摘要尽量由后端聚合输出。
4. **通知事件化**：由订单生命周期事件驱动，不再只是同步调用 webhook。
5. **渐进演进**：V2 不要求一次性推翻现有表结构，可通过新增表和扩展字段实现。

---

## 三、当前后端现状

当前已有：

- `orders` / `order_items`
- `recipes` / `recipe_images`
- `favorites`
- `cook_logs` / `ratings`
- `wishes`
- `wechat.ts` 群 webhook 通知模块

### 已存在问题

1. **订单状态过于简单**：只有 `pending -> confirmed -> completed`
2. **订单详情语义不足**：缺少做菜人、状态轨迹、评论、评价、互动字段
3. **通知为内存队列 + 文本广播**：不可追踪、不可持久化、不可回流
4. **评论对象偏 cook_log**：不适合承接完整订单互动
5. **首页缺少聚合接口**：前端要自己拼多类数据

---

## 四、V2 领域模型升级

## 1. 订单仍作为主聚合根

订单是 V2 的中心对象，其他能力围绕订单扩展：

```text
Order
├─ OrderItems
├─ OrderStatusEvents
├─ OrderComments
├─ OrderReviews
├─ OrderLikes
├─ OrderShares
└─ OrderNotificationDeliveries
```

---

## 2. 订单状态模型升级

### V1 状态

- pending
- confirmed
- completed

### V2 状态建议

- `submitted` 已下单
- `confirmed` 已确认
- `preparing` 制作中
- `completed` 已完成
- `cancelled` 已取消（可选）

### 兼容策略

- 旧数据 `pending` 迁移为 `submitted`
- 当前前端在未完成全面升级前，后端可做状态映射兼容

---

## 3. 新增状态轨迹表

```sql
CREATE TABLE order_status_events (
  id          INTEGER PRIMARY KEY AUTOINCREMENT,
  order_id    INTEGER NOT NULL REFERENCES orders(id) ON DELETE CASCADE,
  from_status TEXT,
  to_status   TEXT NOT NULL,
  operator_id INTEGER REFERENCES users(id),
  note        TEXT,
  created_at  TEXT NOT NULL DEFAULT (datetime('now'))
);
```

### 用途

- 驱动订单详情时间线
- 支撑通知触发
- 支撑后续审计

---

## 4. 新增订单评论表

### 目标

承接“过程评论”，让订单互动不再依赖 cook_log。

```sql
CREATE TABLE order_comments (
  id          INTEGER PRIMARY KEY AUTOINCREMENT,
  order_id    INTEGER NOT NULL REFERENCES orders(id) ON DELETE CASCADE,
  user_id     INTEGER NOT NULL REFERENCES users(id),
  role_type   TEXT NOT NULL, -- requester/cook/member
  content     TEXT NOT NULL,
  created_at  TEXT NOT NULL DEFAULT (datetime('now')),
  updated_at  TEXT NOT NULL DEFAULT (datetime('now'))
);
```

### 说明

- `role_type` 用于前端展示“点菜的人 / 做菜的人 / 家人”身份标签
- 不做 threaded replies 的第一版，先做时间流

---

## 5. 新增订单评价表

### 目标

承接“完成后评价”，与过程评论分离。

```sql
CREATE TABLE order_reviews (
  id             INTEGER PRIMARY KEY AUTOINCREMENT,
  order_id       INTEGER NOT NULL REFERENCES orders(id) ON DELETE CASCADE,
  user_id        INTEGER NOT NULL REFERENCES users(id),
  score          INTEGER CHECK(score >= 1 AND score <= 5),
  taste_score    INTEGER,
  portion_score  INTEGER,
  overall_note   TEXT,
  created_at     TEXT NOT NULL DEFAULT (datetime('now')),
  UNIQUE(order_id, user_id)
);
```

### 与旧 ratings 的关系

- `ratings` 继续保留用于 cook_log 历史能力
- V2 新的订单闭环使用 `order_reviews`

---

## 6. 新增订单点赞 / 分享表

### 点赞

```sql
CREATE TABLE order_likes (
  order_id    INTEGER NOT NULL REFERENCES orders(id) ON DELETE CASCADE,
  user_id     INTEGER NOT NULL REFERENCES users(id),
  created_at  TEXT NOT NULL DEFAULT (datetime('now')),
  PRIMARY KEY (order_id, user_id)
);
```

### 分享记录

```sql
CREATE TABLE order_shares (
  id           INTEGER PRIMARY KEY AUTOINCREMENT,
  order_id      INTEGER NOT NULL REFERENCES orders(id) ON DELETE CASCADE,
  user_id       INTEGER NOT NULL REFERENCES users(id),
  share_type    TEXT NOT NULL, -- order_card/completed_card
  channel       TEXT NOT NULL, -- wechat_moments/link/native
  created_at    TEXT NOT NULL DEFAULT (datetime('now'))
);
```

---

## 7. 增强 orders 表字段

建议给 `orders` 增加：

```sql
ALTER TABLE orders ADD COLUMN cook_user_id INTEGER REFERENCES users(id);
ALTER TABLE orders ADD COLUMN completed_at TEXT;
ALTER TABLE orders ADD COLUMN cancelled_at TEXT;
```

### 说明

- `cook_user_id`：标识当前负责做菜的人
- `completed_at`：已完成时间
- `cancelled_at`：取消时间

## 五、图片上传链路改造

## 1. 当前问题

当前 recipe 创建成功后立即上传图片，但后端没有“图片保存状态语义”，前端也只能把异常粗暴显示为整体失败。

## 2. 后端 V2 原则

- 保持 recipe 与 image 资源解耦
- recipe 创建成功即视为业务成功
- image 保存失败只影响附件状态，不回滚 recipe 本体

## 3. API 层要求

继续保留：

- `GET /api/upload/presign`
- `POST /api/recipes/:id/images`

新增建议：

- `DELETE /api/images/:id`
- `POST /api/recipes/:id/images/retry`（可选，若服务端需要记录重试）

## 六、API 设计方案

## 1. 订单相关 API

### 查询订单列表

```http
GET /api/orders?status=submitted|confirmed|preparing|completed
```

### 查询订单详情

```http
GET /api/orders/:id
```

V2 返回结构建议包含：

- order 基础字段
- requester 信息
- cook 信息
- items
- status timeline
- comments summary / comments
- reviews summary / reviews
- like/favorite/share 状态

### 创建订单

```http
POST /api/orders
```

创建时自动：

1. 写入 `orders`
2. 写入 `order_items`
3. 写入首条 `order_status_events`（submitted）
4. 生成通知事件

### 更新订单状态

```http
PUT /api/orders/:id/status
```

后端行为：

1. 校验合法流转
2. 更新 `orders.status`
3. 写入 `order_status_events`
4. 写入通知事件

---

## 2. 订单评论 API

```http
GET /api/orders/:id/comments
POST /api/orders/:id/comments
```

### 规则

- 仅家庭成员可见
- 创建者、做菜人、家庭成员都可评论
- 评论按时间正序或倒序统一输出

---

## 3. 订单评价 API

```http
GET /api/orders/:id/reviews
POST /api/orders/:id/reviews
```

### 规则

- 仅完成订单可评价
- 每个用户每单只能评价一次
- 后续可允许补充/修改

---

## 4. 点赞 / 分享 API

```http
POST /api/orders/:id/like
DELETE /api/orders/:id/like
POST /api/orders/:id/share
GET /api/orders/:id/share-card
```

### 说明

- `share-card` 返回分享卡渲染所需数据
- `share` 记录一次分享动作，用于统计与成就

---

## 5. 首页聚合 API

```http
GET /api/home/summary
```

### 返回内容

- 今日推荐菜
- 常点菜
- 最近订单动态

### 原因

## 6. 我的页聚合 API

### 返回内容

- 我的订单统计

- 我的评论数
- 我点过的 / 我做过的摘要
- 家庭成员概要

## 七、通知系统重构

## 1. 当前问题

当前 `backend/src/lib/wechat.ts`：

- 是内存队列
- 只发文本
- 进程重启丢消息
- 不记录投递结果
- 不支持订单详情 deeplink

## 2. V2 目标

把通知升级为 **订单事件驱动的可靠投递系统**。

## 3. 最小可行方案

### 新增通知事件表

```sql
CREATE TABLE notification_events (
  id            INTEGER PRIMARY KEY AUTOINCREMENT,
  family_id      INTEGER NOT NULL REFERENCES families(id),
  event_type     TEXT NOT NULL, -- order_created/order_confirmed/order_preparing/order_completed
  entity_type    TEXT NOT NULL, -- order
  entity_id      INTEGER NOT NULL,
  payload        TEXT NOT NULL, -- JSON
  status         TEXT NOT NULL DEFAULT 'pending', -- pending/sent/failed
  created_at     TEXT NOT NULL DEFAULT (datetime('now')),
  sent_at        TEXT,
  last_error     TEXT
);
```

### 新增投递记录表

```sql
CREATE TABLE notification_deliveries (
  id              INTEGER PRIMARY KEY AUTOINCREMENT,
  notification_id INTEGER NOT NULL REFERENCES notification_events(id) ON DELETE CASCADE,
  target_user_id  INTEGER REFERENCES users(id),
  channel         TEXT NOT NULL, -- wecom_group/wecom_app/in_app
  status          TEXT NOT NULL DEFAULT 'pending',
  attempt_count   INTEGER NOT NULL DEFAULT 0,
  last_error      TEXT,
  created_at      TEXT NOT NULL DEFAULT (datetime('now')),
  updated_at      TEXT NOT NULL DEFAULT (datetime('now'))
);
```

## 4. 触发点

- 订单创建
- 订单确认
- 制作中
- 已完成
- 新评论
- 新评价（P1）

## 5. 实现策略

P0 可先保留群 webhook，但必须：

1. 从同步调用改为落库 + worker/cron 拉取发送
2. 记录失败原因
3. payload 中保留 `orderId`
4. 为后续 deeplink 留出扩展字段

- `order_comments`
- `order_likes`

## 九、查询与性能策略

## 1. 订单详情聚合查询

`GET /api/orders/:id` 应尽量由后端一次聚合返回，而不是让前端请求 5-6 个接口。

## 2. 首页摘要聚合

首页适合单接口聚合，避免前端多接口 waterfall。

## 3. SQLite 约束

保持：

- WAL
- busy_timeout
- foreign_keys=ON

并继续避免：

- N+1 查询
- 循环内逐条 select

---

## 十、迁移策略

## 1. 渐进式迁移

### Phase 1

- 扩展订单状态
- 新增 `order_status_events`
- 新增 `/orders/:id` 详情聚合返回

### Phase 2

- 新增 `order_comments`
- 新增 `order_reviews`
- 新增 `order_likes` / `order_shares`

### Phase 3

- 新增 `notification_events` / `notification_deliveries`
- 新增 `home/summary` / `profile/summary` / `achievements` 聚合接口

## 2. 兼容要求

- 旧 `ratings`、`cook_logs` 继续保留
- 旧 `pending/confirmed/completed` 接口短期可兼容映射
- 前端未切完前，后端不要一次性破坏既有路由

---

## 十一、模块拆分建议

## 1. 路由模块

- `routes/orders.ts` 订单列表/详情/创建/状态流转
- `routes/order-comments.ts`
- `routes/order-reviews.ts`
- `routes/order-interactions.ts`（like/share）
- `routes/home.ts`
- `routes/profile.ts`
- `routes/achievements.ts`

## 2. 领域服务层

- `services/order-service.ts`
- `services/order-timeline-service.ts`
- `services/notification-service.ts`
- `services/home-summary-service.ts`
- `services/achievement-service.ts`

## 3. 通知层

- `lib/notification-queue.ts`
- `lib/wecom.ts`
- `workers/notification-dispatch.ts`（若采用定时拉取）

---

## 十二、约束与禁止项

1. 不把通知可靠性继续建立在纯内存队列上。
2. 不把订单评论继续强行塞进 cook_log/ratings 语义里。
3. 不让首页依赖前端发起过多串行请求。
4. 不破坏 familyId 访问边界。
5. 不用“为了简单”省略订单状态轨迹表。

---

## 十三、完成标准

当 V2 后端核心能力落地后，应满足：

1. `GET /api/orders/:id` 能返回完整订单详情聚合数据。
2. 订单状态变更会留下可查询的状态时间线。
3. 订单支持评论与完成后评价。
4. 通知具备最小可追踪能力，而非纯同步广播。
5. 首页、我的页、成就页具备聚合接口基础。
