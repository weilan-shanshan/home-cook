# 01-入口与路由：任务分发逻辑 (Intake & Routing)

> **核心逻辑**: 将非结构化的原始需求转化为结构化、可调度的任务流。

## 1. 需求摄入 (Intake)

所有新需求首先由 **Intake Agent** 处理，按照 [需求简报模板](./templates/intake-brief-template.md) 进行结构化：

- **业务目标 (Business Goal)**: 用一句话描述用户真正想要解决的问题。
- **技术约束 (Technical Constraints)**: 现有技术栈、安全性要求、性能基准。
- **相关上下文 (Context Reference)**: 涉及的现有代码、配置文件或 Notepad 记录。

## 2. 路由与分类 (Routing & Categorization)

摄入后，将需求路由至不同类型的任务执行器。任务分类决定了执行策略：

| 任务分类 (Category) | 特征描述 | 适用场景 |
| :--- | :--- | :--- |
| **Quick** | 逻辑单一，可单次对话完成，文件改动少 | 简单 Bugfix, 文案修改, 配置更新 |
| **Visual-Engineering**| 涉及 UI、样式、响应式，需浏览器验证 | 页面组件开发, 样式优化, 交互实现 |
| **Deep** | 业务逻辑复杂，跨多个核心模块，高安全性要求 | 认证系统, 复杂算法, 核心数据结构修改 |
| **Infrastructure** | 涉及 CI/CD, 部署, 数据库迁移, 环境配置 | 部署脚本, 备份方案, 容器化 |
| **Audit/Review** | 交叉验证, 安全扫描, 质量合规检查 | 交付前质量把控, 安全审计 |

## 3. 路由规则 (Routing Rules)

- **并发优先**: 尽量将任务拆解为可并行的（Wave 结构）。
- **相似聚合**: 相似的功能点（如 CRUD）应聚合给同一个 Agent 以保持上下文连贯性。
- **技能匹配**: 
  - `visual-engineering` 优先分配给加载了 `frontend-ui-ux` 技能的 Agent。
  - `Infrastructure` 优先分配给具备系统运维背景的 Agent。

## 4. 记忆 / 检索路由 (Memory Routing)

在 Intake 阶段，除了给任务分类，也要给“信息”分类：

| 信息类型 | 应进入哪里 | 判断标准 |
|---|---|---|
| 长期稳定项目事实 | `MEMORY.md` | 高复用、跨任务稳定、每轮大概率会用到 |
| 长期稳定用户偏好 | `USER.md` | 跨主题稳定，不会随当前任务频繁变化 |
| 之前聊过的场景 | `session_search` | 需要回忆“发生过什么”而不是“长期事实是什么” |
| 可复用流程 | Skills | 这是“怎么做”的套路，而非具体任务内容 |
| 深层用户画像 | Honcho（可选） | 只有当其明显提升协作质量时才启用 |

## 5. 防错边界 (Routing Guardrails)

- **禁止路由模糊需求**: 若需求简报中缺少 3 个以上关键要素，禁止分发任务，必须打回 Intake 阶段补全。
- **强制依赖校验**: 任何任务在路由时必须明确其 `deps` 列表，防止提前分发导致的依赖缺失失败。
- **负载均衡**: 同一 Wave 任务数不建议超过 10 个，避免上下文污染。
- **禁止把情景回忆写进热记忆**: “上周讨论过什么”应走 `session_search`，不应挤进 `MEMORY.md`。
- **禁止把 Skill 当知识库全文注入**: Skills 必须索引化，命中后再完整加载。

---
*通过精细化的摄入与路由，将复杂需求在进入执行层前完成"降噪"与"定向"。*
