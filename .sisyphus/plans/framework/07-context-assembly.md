# 07-上下文装配：提示词顺序与缓存策略 (Context Assembly)

> **核心逻辑**: 上下文不仅是“放什么”，更是“按什么顺序放”。顺序稳定，缓存才稳定；热信息靠前，检索信息靠后。

## 1. 推荐装配顺序

参考 Hermes 的上下文设计，本仓库建议采用如下装配顺序：

```text
[0] 默认 Agent 身份
[1] 工具感知的行为引导
[2] Honcho / 用户深层画像（可选）
[3] MEMORY.md 冻结快照
[4] USER.md 冻结快照
[5] Skills 索引
[6] 仓库上下文文件（AGENTS.md / plans / rules）
[7] 当前任务相关信息
[8] 可选的会话摘要 / 检索结果
[9] 对话历史
[10] 当前用户消息
```

### 顺序原则

- **越稳定越靠前**: Agent 身份、行为约束、热记忆放在前面。
- **越常用越靠前**: 高频事实放进固定前缀，减少重复计算。
- **越大越靠后**: 会话摘要、会话历史、当前消息天然波动，放在后面。
- **越偶发越按需**: 可检索的信息不应挤占固定提示词空间。

## 2. 提示词缓存原则

Hermes 的核心启发不是“多存”，而是：

> **提示词尽量不变，能缓存的都缓存，剩下的按需检索。**

这意味着：

- `MEMORY.md` / `USER.md` 需要小、稳、精，并构成尽可能固定的前缀。
- 不要把任务进度、临时排查过程、一次性上下文放进热记忆。
- Skills 只注入索引；完整内容在命中后加载。
- “我们上周讨论过这个吗”这类问题，优先走 `session_search`，并将检索结果放在稳定前缀之后。

## 3. 何时命中哪一层

| 问题类型 | 优先层 | 例子 |
|---|---|---|
| 长期稳定事实 | `MEMORY.md` | 项目路径、固定技术栈、长期环境约束 |
| 用户长期偏好 | `USER.md` | 喜欢中文、偏简洁、偏先实现后解释 |
| 情景回忆 | `session_search` | “我们上次讨论过的部署方案是哪个？” |
| 可复用方法 | Skills | “AJX 项目怎么构建并真机调试？” |
| 深层人物画像 | Honcho（可选） | 跨主题稳定身份、协作习惯、决策风格 |

## 4. 防错边界

- **禁止把日志当记忆**: 会话日志属于 `session_search`，不属于 `MEMORY.md`。
- **禁止把临时状态写入 USER**: 用户一时的任务偏好，不等于长期画像。
- **禁止 Skill 全量常驻**: Skill 必须索引化，避免固定前缀膨胀。
- **禁止频繁改写前缀顺序**: 顺序抖动会伤害缓存命中率。

## 5. 交付建议

在本仓库中，建议把这套上下文装配规范体现在：

1. `.sisyphus/plans/framework/` 的规则文档中。
2. `MEMORY.template.md` / `USER.template.md` / `skills-index-template.md` 中。
3. 后续 Wave 7 试点任务的证据与复盘中。
