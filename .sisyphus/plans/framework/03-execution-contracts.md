# 03-执行契约：DoD 与 Guardrails (Execution Contracts)

> **核心逻辑**: 在执行任务的过程中，如何通过定义行为边界来确保高质量的交付。

## 1. 完成定义 (Definition of Done, DoD)

只有满足以下所有条件的任务才可标记为 `done`：

- **功能符合预期**: 完全实现 AC (Acceptance Criteria) 中的所有点。
- **无回归风险**: 运行现有测试套件，确保没有破坏现有功能。
- **通过 Lint/Typecheck**: 无 ESLint 错误，无 TypeScript 类型错误。
- **代码整洁**: 符合项目既定命名规范，无冗余代码或 `console.log`。
- **证据已持久化**: 按照 [04-证据政策](./04-evidence-policy.md) 提交证据。
- **记忆已同步**: 决策记录在 `decisions.md`，坑点记录在 `learnings.md`。

## 2. 行为边界 (Guardrails)

Agent 在执行任务时必须严格遵守的"红线"：

### 技术红线
- **禁止硬编码**: 密钥、Webhook 必须来自 `.env` 或环境变量。
- **禁止 `as any`**: 除非获得显式授权，否则禁止跳过类型检查。
- **禁止忽略错误**: 异步调用必须处理 `reject` 或 `try-catch`。
- **禁止 IDOR**: 所有数据查询必须包含 `family_id` 过滤。
- **禁止 Base64 上传**: 图片必须使用预签名 URL。

### 协作红线
- **禁止默默失败**: 遇到阻碍必须立即报告，并描述已尝试的方案。
- **禁止删除用户代码**: 除非任务明确要求重构，否则严禁随意删除或重写用户已写好的逻辑。
- **禁止偏离计划**: 任务执行过程中若需增加功能，必须先报备。
- **禁止空闲等待**: 计划外的空闲应主动检查 Notepad 或相关 Issue。

## 3. 执行工作流 (Execution Workflow)

1. **环境初始化**: 检查依赖，读取最新全局配置文件。
2. **渐进式实现**: 先搭建结构（如接口定义、UI 骨架），再填充逻辑。
3. **实时验证**: 每完成一个子模块，立即运行 `QA Scenarios`。
4. **日志输出**: 实时向控制台/用户反馈当前的进展和发现的问题。
5. **任务收尾**: 运行全局 Lint/Build/Test。

## 4. 故障处理 (Error Recovery)

- **自愈机制**: 若编译失败，先检查 imports、依赖版本和 tsconfig。
- **分阶段提交**: 鼓励在大型任务中进行分阶段逻辑提交（如果 git 允许）。
- **回滚准则**: 若一次性尝试失败，应及时清理脏代码，回退到任务开始前的状态重新开始。

---
*规范的执行行为是跨 Agent 协作稳定性的基石。*
