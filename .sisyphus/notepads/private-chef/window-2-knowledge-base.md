# Window 2 Implementation Knowledge Base

This document summarizes the key implementation details, architectural decisions, and technical contracts established during Window 2 of the Private Chef project. It is optimized for AI agents to quickly understand the current state of the codebase.

## Shared Conventions & Architecture

- **Auth Strategy**: Cookie-based session management. The `session` cookie (HttpOnly, Secure, sameSite: 'Strict') is handled by `backend/src/lib/auth.ts` and `backend/src/middleware/auth.ts`. 
- **Database**: SQLite (better-sqlite3) with Drizzle ORM. Pragmas (WAL, foreign_keys, etc.) are set in `backend/src/db/index.ts`.
- **Family Isolation**: All business logic (recipes, tags, orders, etc.) MUST filter by `familyId`, which is injected into the Hono context (`c.get('familyId')`) by the `authMiddleware`.
- **Frontend Networking**: ❌ No bare `fetch`. Use the typed `apiFetch` wrapper in `frontend/src/lib/api.ts` which handles `credentials: 'include'` and base URL prefixing.
- **UI Style**: Apple-style "glassmorphism" using the `.glass-card` and `.glass-nav` Tailwind classes defined in `frontend/src/styles/globals.css`.

## Task Summaries (Window 2)

### T02: DB Schema & Migration
- **Files**: `backend/src/db/schema.ts`, `backend/src/db/index.ts`
- **Key Constraints**: 
  - `order_items.recipe_id` uses `ON DELETE RESTRICT` to prevent deleting recipes with active orders.
  - `ratings.score` has a `CHECK(score >= 1 AND score <= 5)` constraint.
  - `tags` has a `UNIQUE(family_id, name)` constraint.
- **Gotchas**: SQLite foreign keys are disabled by default; `pragma('foreign_keys = ON')` in `db/index.ts` is critical.

### T08: Tags System
- **Files**: `backend/src/routes/tags.ts`
- **Endpoints**: `GET /api/tags`, `POST /api/tags`, `DELETE /api/tags/:id`.
- **Behavior**: Tags are scoped to families. Creating a duplicate tag name within the same family returns `409 Conflict`.

### T09: Image Upload (COS)
- **Files**: `backend/src/routes/images.ts`, `backend/src/lib/cos.ts`
- **Flow**: 
  1. Frontend calls `GET /api/upload/presign` to get a COS PUT URL.
  2. Frontend uploads directly to COS via `XMLHttpRequest` (for progress).
  3. Frontend calls `POST /api/recipes/:id/images` to link the COS URL to the recipe.
- **Decisions**: Backend does NOT touch image bytes. `DELETE /api/images/:id` only removes the DB record; COS file cleanup is deferred to a background worker (Phase 2).

### T10: Orders System
- **Files**: `backend/src/routes/orders.ts`
- **Endpoints**: `GET /api/orders`, `POST /api/orders`, `PUT /api/orders/:id/status`.
- **Logic**: 
  - **Transactions**: `POST` uses a database transaction to insert the order and all items atomically.
  - **Status Flow**: Strict one-way progression: `pending` -> `confirmed` -> `completed`. Attempts to "roll back" status return `400 Bad Request`.
  - **Notifications**: Successful creation triggers `notifyNewOrder()` (T05) for WeChat/Enterprise WeChat alerts.

### T16: Recipe Pages (Frontend)
- **Files**: `frontend/src/pages/home/Home.tsx`, `frontend/src/pages/recipe/RecipeDetail.tsx`, `frontend/src/pages/recipe/RecipeForm.tsx`, `frontend/src/lib/upload.ts`
- **Implementation**:
  - **Infinite Scroll**: Uses `useInfiniteQuery` for the home page recipe list.
  - **Upload Tool**: `frontend/src/lib/upload.ts` uses `browser-image-compression` (≤1MB) and `XMLHttpRequest`.
  - **Lifecycle**: MUST call `URL.revokeObjectURL` after image previews are no longer needed to prevent memory leaks.
- **Gotchas**: Full-width search bars avoid `backdrop-filter` for performance reasons.

### T17: Order Pages (Frontend)
- **Files**: `frontend/src/pages/order/OrderCreate.tsx`, `frontend/src/pages/order/OrderList.tsx`
- **Logic**: 
  - `OrderCreate` supports multi-select recipes and quantity adjustments.
  - `OrderList` groups orders by date for a better "meal plan" view.

### T22: PWA Configuration
- **Files**: `frontend/vite.config.ts`
- **Strategies**:
  - `/api/*`: `NetworkFirst` (ensure fresh data when online).
  - Assets (`js`, `css`): `CacheFirst`.
  - Images: `StaleWhileRevalidate`.
- **Note**: The web manifest is generated by `vite-plugin-pwa` during build; it is not a manually maintained source file.
- **Node Version**: The `vite-plugin-pwa` version used is compatible with Node 18.17.1.

### T23: Critical Path Tests
- **Files**: `backend/src/__tests__/auth.test.ts`, `backend/src/__tests__/orders.test.ts`, `backend/src/__tests__/helpers.ts`, `backend/src/app.ts`
- **Strategy**: Uses `vitest` with a temporary SQLite file created per test context/case.
- **Isolation & Setup**:
  - Sets `process.env.DATABASE_PATH` to a unique temp path before dynamic imports.
  - Calls `vi.resetModules()` to ensure the DB singleton re-initializes for each test context.
  - `src/app.ts` was introduced specifically to export the Hono app instance without starting the server, allowing tests to import it without side effects from `src/index.ts`.
  - **Cleanup**: Must close the `sqlite` handle and explicitly delete the temporary database file along with any `-wal` and `-shm` sidecars to prevent leakage.
  - **Node 18**: Manually injects `globalThis.crypto = webcrypto` for environments lacking global crypto in tests.

## Extending the System

- **Adding a Table**: Update `backend/src/db/schema.ts`, run `npx drizzle-kit generate`, and ensure the new table is included in `backend/src/db/schema.ts` exports for the `AppType`.
- **Adding a Route**: Create the route in `backend/src/routes/` and mount it in `backend/src/app.ts`. Follow the existing pattern for mounting routes on the `app` instance.
- **Frontend Components**: Use `shadcn/ui` components. Apply `.glass-card` for content containers to match the Apple-style aesthetic.
- **Testing**: Follow the pattern in `helpers.ts` to ensure DB isolation and proper session cookie injection in `supertest`/`vitest` requests.
