# 框架学习记录 (Learnings)

> **版本**: 1.0  
> **分类**: Framework  
> **日期**: 2026-04-13

## L01: 证据驱动对任务完结的重要性
- **现象**: 在 T29-T34 的初期规划中，发现部分任务仅通过 Agent 文字描述即可宣称完成，缺乏客观的验证手段。
- **原因**: 现有的任务详单未强制链接物理证据。
- **学习点**: 在 `04-evidence-policy.md` 中强制要求所有 QA Scenarios 指向具体的物理证据文件，并定义了不包含物理证据的交付为 "幻觉交付"。
- **应用范围**: 框架的所有任务详单和模板。

## L02: Wave 7 引入后的依赖管理优化
- **现象**: 随着任务数超过 30 个，手动维护依赖关系图变得更加困难。
- **原因**: 线性依赖关系的局限性。
- **学习点**: 采用 Wave 结构不仅是并行的需要，更是逻辑隔离的需要。Wave 7 作为一个全新的 "框架 Wave"，其依赖于前面的 Wave 完成但逻辑上相对独立。
- **应用范围**: `plans/tasks/README.md`。

## L03: 中文 Markdown 在不同 Agent 间的解析一致性
- **现象**: 不同厂商的 AI 模型对中文标题和列表的解析存在微小偏差。
- **原因**: Markdown 渲染引擎和分词算法的不同。
- **学习点**: 在模板中使用清晰的 `## {标题}` 结构和 `| 表格 |` 语法，避免使用过于复杂的嵌套列表，有助于提升不同 Agent 间的信息提取准确度。
- **应用范围**: 框架的所有核心规范。

## L04: 热记忆应该是“精选集”而不是“对话摘录”
- **现象**: 如果把任务进度、临时排障、历史讨论都写入热记忆，提示词前缀会迅速膨胀。
- **原因**: 把持久事实、情景回忆、程序性方法混成了一层。
- **学习点**: 高频稳定事实进入 `MEMORY.md` / `USER.md`，历史上下文交给 `session_search`，方法论交给 Skills。
- **应用范围**: `05-memory-loop.md`, `07-context-assembly.md`。

## L05: 第一轮 pilot 应优先验证“闭环”而不是“难度”
- **现象**: 复杂业务任务虽然更有挑战，但不利于判断问题出在框架还是业务本身。
- **原因**: 首次试点的目标应是证明 intake、evidence、review、memory sync 能走通。
- **学习点**: 用小而真实的文档基础设施任务做第一轮 pilot，更适合给框架定基线。
- **应用范围**: `t34-framework-pilot.md`, 后续 pilot 任务设计。
