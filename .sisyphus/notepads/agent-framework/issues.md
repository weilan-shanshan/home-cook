# 框架问题记录 (Issues)

> **版本**: 1.0  
> **分类**: Framework  
> **日期**: 2026-04-13

## I01: 自动化评审 Agent (Oracle) 的实际可用性限制 (OPEN)
- **描述**: 虽然框架在 `06-review-gates.md` 中定义了由 `oracle` Agent 进行交叉审计，但当前环境中可能无法随时调用外部专门审计模型。
- **潜在风险**: 评审过程可能退化为当前执行 Agent 的自检。
- **缓解对策**: 在 T34 试点任务中，模拟 Logic-Review 流程，手动检查 AC 达成度，并记录在评审报告模板中。
- **负责人**: Sisyphus-Junior

## I02: Notepad 的增量同步并发冲突风险 (OPEN)
- **描述**: 当多个执行窗口同时尝试 `Edit` 同一个 `decisions.md` 时，可能存在覆盖风险。
- **潜在风险**: 丢失关键决策。
- **缓解对策**: 在任务间通过 Wave 隔离来减少并发写入同一个 Notepad 的概率。建议每个 Wave 完成后进行一次全局同步。
- **负责人**: Sisyphus-Junior

## I03: 证据文件的清理与索引机制缺失 (OPEN)
- **描述**: 随着 T 任务数量增加，`.sisyphus/evidence/` 下的文件将超过 100 个，缺乏自动化的清理和快速索引机制。
- **潜在风险**: 证据文件膨胀导致上下文读取变慢。
- **缓解对策**: 在 T34 后续演进中考虑增加 `evidence/README.md` 的索引机制，并按照 Wave 进行子目录划分。
- **负责人**: Sisyphus-Junior

## I04: 热记忆预算漂移风险 (OPEN)
- **描述**: 当 `MEMORY.md` / `USER.md` 缺少定期裁剪机制时，很容易从高命中率摘要退化为低命中率仓库说明书。
- **潜在风险**: 固定提示词前缀变大、缓存命中下降、真正重要信息被稀释。
- **缓解对策**: 对热记忆采用字符预算和 `replace/remove` 优先策略，定期执行瘦身审查。
- **负责人**: Sisyphus

## I05: T34 仍缺少“中高复杂度代码任务”级别的第二轮试点 (OPEN)
- **描述**: 当前 pilot 已验证最小闭环，但尚未覆盖多文件代码实现型任务。
- **潜在风险**: 框架在真实编码任务上的摩擦成本仍未充分暴露。
- **缓解对策**: 后续选择一个跨 2-3 个文件的小型实现任务做 second pilot。
- **负责人**: Sisyphus
